This guide walks you through installing gitagent, scaffolding a new agent, validating it, and running it locally with Claude Code.

## Prerequisites

- Node.js 18 or later
- npm 7 or later

For the `run` step you will also need the [Claude Code CLI](https://docs.anthropic.com/en/docs/claude-code) installed globally.

---

<Steps>

<Step title="Install gitagent">

Install the CLI globally from npm:

```bash
npm install -g gitagent
```

Verify the installation:

```bash
gitagent --version
```

You should see output like `0.1.7`. If not, see the [Installation](/installation) page for troubleshooting.

</Step>

<Step title="Scaffold a new agent">

Create a new agent directory with the `standard` template. This gives you a solid starting point with an identity, rules, a skills directory, and a knowledge index — all ready to customize.

<CodeGroup>

```bash minimal
gitagent init --template minimal
```

```bash standard (recommended)
gitagent init --template standard
```

```bash full (with compliance)
gitagent init --template full --dir ./my-agent
```

</CodeGroup>

The three templates differ in what they create:

| Template | Files created |
|----------|---------------|
| `minimal` | `agent.yaml`, `SOUL.md` |
| `standard` | `agent.yaml`, `SOUL.md`, `RULES.md`, `AGENTS.md`, `skills/`, `knowledge/`, `tools/` |
| `full` | Everything in `standard` plus `memory/`, `hooks/`, `examples/`, `agents/`, `compliance/`, `config/`, `.gitignore` |

After running `gitagent init --template standard` you will see:

```
✔ Created agent.yaml
✔ Created SOUL.md
✔ Created RULES.md
✔ Created AGENTS.md
✔ Created skills/example-skill/SKILL.md
✔ Created knowledge/index.yaml

Agent scaffolded at /your/directory
Next steps:
  1. Edit agent.yaml with your agent details
  2. Write your SOUL.md identity
  3. Run `gitagent validate` to check your configuration
```

<Tip>
  Only two files are strictly required: `agent.yaml` (the manifest) and `SOUL.md` (the identity). Everything else is optional — add what you need, leave the rest.
</Tip>

</Step>

<Step title="Edit the core files">

Open `agent.yaml` and fill in your agent's identity. The minimal required fields are `spec_version`, `name`, `version`, and `description`:

```yaml agent.yaml
spec_version: "0.1.0"
name: my-agent
version: 0.1.0
description: A helpful assistant agent
author: ""
license: MIT
model:
  preferred: claude-sonnet-4-5-20250929
  constraints:
    temperature: 0.3
    max_tokens: 4096
skills: []
tools: []
runtime:
  max_turns: 30
  timeout: 120
tags: []
```

Then open `SOUL.md` and write your agent's identity and values:

```markdown SOUL.md
# Soul

## Core Identity
I am a helpful assistant specializing in [your domain].

## Communication Style
Clear, concise, and professional. I adapt my tone to the context.

## Values & Principles
- Accuracy over speed
- Transparency in reasoning
- Helpfulness without overstepping
```

</Step>

<Step title="Validate the agent">

Run the validator to check your configuration against the gitagent specification:

```bash
gitagent validate
```

The validator checks:

- **`agent.yaml`** — JSON schema validation; referenced skills, tools, and sub-agents all exist
- **`SOUL.md`** — Exists, is not empty, has substantive content
- **`skills/`** — Valid SKILL.md frontmatter, name matches directory, description length, instruction size
- **`hooks/hooks.yaml`** — Valid YAML schema, referenced scripts exist
- **`tools/*.yaml`** — Valid tool schema

To also check regulatory compliance rules:

```bash
gitagent validate --compliance
```

A passing result looks like:

```
Validating gitagent
Directory: /your/directory
────────────────────────────────────────
✔ agent.yaml — valid
✔ SOUL.md — valid
✔ skills/ — valid
────────────────────────────────────────
✔ Validation passed (0 warnings)
```

<Note>
  When you run `gitagent init`, the scaffolded files pass validation immediately. Edit them to add your own content, then validate again.
</Note>

</Step>

<Step title="Inspect the agent">

View a formatted summary of your agent configuration:

```bash
gitagent info
```

This shows the agent name, version, description, model preferences, skills, tools, sub-agents, runtime config, compliance settings, and a preview of `SOUL.md`.

To inspect an agent in a specific directory:

```bash
gitagent info -d ./examples/standard
```

</Step>

<Step title="Export to a system prompt">

Export your agent definition to a portable system prompt you can use with any LLM:

```bash
gitagent export --format system-prompt
```

To save the output to a file instead of printing it:

```bash
gitagent export --format system-prompt --output system-prompt.md
```

Other useful export targets:

<CodeGroup>

```bash Claude Code
gitagent export --format claude-code --output CLAUDE.md
```

```bash OpenAI Python SDK
gitagent export --format openai --output agent.py
```

```bash CrewAI YAML
gitagent export --format crewai --output crew.yaml
```

</CodeGroup>

</Step>

<Step title="Run the agent">

Run your agent locally with the Claude Code adapter (the default):

```bash
gitagent run -d .
```

This generates a system prompt from your agent definition and launches Claude Code with the appropriate flags — model preferences, tool permissions, runtime settings, and hook mappings — all derived from your `agent.yaml`.

To run a one-shot prompt without interactive mode:

```bash
gitagent run -d . -p "Summarize this project"
```

To run an agent directly from a public git repository:

```bash
gitagent run -r https://github.com/user/my-agent -p "Hello"
```

To run with a different adapter:

```bash
# GitHub Models (requires GITHUB_TOKEN)
gitagent run -d . -a github -p "Review my code"

# Lyzr Studio (requires LYZR_API_KEY)
gitagent run -d . -a lyzr -p "Hello"
```

<Warning>
  The `claude` adapter (default) requires the Claude Code CLI to be installed. Run `claude --version` to verify. Other adapters have their own requirements — see [Adapters](/adapters/overview) for details.
</Warning>

</Step>

</Steps>

## What you just built

Your agent directory is now a **git-native agent definition**: version-controllable, portable across frameworks, and ready for CI/CD with `gitagent validate` on every push.

The `standard` template gives you:

```
my-agent/
├── agent.yaml                  # Manifest — identity, model, skills, runtime
├── SOUL.md                     # Identity, personality, values
├── RULES.md                    # Hard constraints and safety boundaries
├── AGENTS.md                   # Fallback instructions for any framework
├── skills/
│   └── example-skill/
│       └── SKILL.md            # Reusable capability module
├── tools/                      # MCP-compatible tool definitions
└── knowledge/
    └── index.yaml              # Reference documents index
```
