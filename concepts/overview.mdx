GitAgent defines a portable, version-controlled format for AI agents. The fundamental idea is simple: **a git repository is the agent**. Every file in the repo is part of the agent definition — identity, rules, capabilities, compliance, memory. Clone the repo, get the agent.

This means every tool git gives you — commits, branches, diffs, tags, PRs, blame — works directly on your agent. There's no separate configuration server, no proprietary format to export to, no vendor lock-in.

## The mental model: files are behavior

Each file in a gitagent repository describes a specific dimension of agent behavior:

| File / directory | What it controls |
|-----------------|------------------|
| `agent.yaml` | Identity, model, capabilities, compliance — the manifest |
| `SOUL.md` | Personality, tone, values, domain expertise |
| `RULES.md` | Hard constraints — what the agent must always or never do |
| `DUTIES.md` | Segregation of duties policy and role boundaries |
| `skills/` | Reusable capability modules the agent can activate |
| `tools/` | MCP-compatible tool definitions |
| `knowledge/` | Reference documents the agent consults at runtime |
| `memory/` | Persistent cross-session state |
| `workflows/` | Multi-step procedures and playbooks |
| `hooks/` | Lifecycle event handlers (bootstrap, teardown, audit) |
| `agents/` | Sub-agent definitions (recursive structure) |
| `compliance/` | Regulatory artifacts for audits and validation |

Only two files are required to define a valid agent: **`agent.yaml`** and **`SOUL.md`**. Everything else is optional — add what you need, skip what you don't.

## How it fits together

<Steps>
  <Step title="Define the agent">
    Create `agent.yaml` and `SOUL.md`. These two files are the minimum. `agent.yaml` declares the manifest — name, version, model preferences, skills, tools, and compliance configuration. `SOUL.md` defines who the agent is: its personality, values, and communication style.
  </Step>
  <Step title="Add capabilities">
    Drop skills into `skills/`, tool schemas into `tools/`, and reference documents into `knowledge/`. Reference skills and tools by name in `agent.yaml` to activate them. Each skill is an independent module with its own instructions and allowed tools.
  </Step>
  <Step title="Validate">
    Run `gitagent validate` to check the manifest schema, verify all referenced skills and tools exist, and (with `--compliance`) validate against regulatory requirements. Validation can run in CI on every push.
  </Step>
  <Step title="Run or export">
    Use `gitagent run` to launch the agent directly with a supported adapter (Claude Code, OpenAI, Lyzr, GitHub Models, and more). Or use `gitagent export` to generate a format-specific artifact — a system prompt, a Python file, a YAML config — for any downstream framework.
  </Step>
</Steps>

## Why git?

Git solves problems for agents that every other approach has to reinvent from scratch:

- **Versioning** — every `git commit` is a snapshot of your agent. Roll back a bad prompt with `git revert`.
- **Branching** — experiment with new skills or personalities on a branch without affecting production.
- **Diffing** — `git diff` shows exactly what changed between agent versions, line by line.
- **Collaboration** — pull requests, code review, and branch protection work on agent definitions the same way they work on code.
- **Releases** — `git tag v1.2.0` pins a stable agent version. Production can track a tag; staging can track `main`.
- **Audit trail** — `git blame` traces every line of every file to who changed it and when — out of the box.

</Note>
